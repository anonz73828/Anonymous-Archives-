Ettercap

Ettercap is a free and open-source comprehensive suite designed for man-in-the-middle (MITM) attacks on local area networks. It supports sniffing of live connections, content filtering on the fly, active and passive dissection of numerous protocols (including encrypted ones), ARP poisoning, DNS spoofing, and various features for network and host analysis.

Uses

Ettercap is utilized for network security auditing, protocol analysis, interception of traffic, demonstration of MITM vulnerabilities, and credential capture in controlled environments. Example: Performing ARP spoofing to position oneself between two hosts and inspect or modify their communication during authorized penetration testing.

Installation in Termux

Ettercap is not available as a standard package in Termux repositories. Compilation from source is possible but requires dependencies and may face permission or library issues. A common approach uses community repositories (e.g., tur-repo) or manual build. For text-only mode (recommended in Termux due to limited GUI support):

pkg update && pkg upgrade -y
pkg install tur-repo -y  # If tur-repo is available; check current status
pkg install ettercap-common ettercap-text-only -y
# Alternatively, compile from source:
pkg install git cmake build-essential libpcap libnet libpcre flex bison -y
git clone https://github.com/Ettercap/ettercap.git
cd ettercap
mkdir build && cd build
cmake ..
make
# Run as: ./bin/ettercap -T (text mode)

Common issues: Permission denied on config files—edit /etc/ettercap/etter.conf with proper access; GUI mode requires X11/VNC setup (similar to Wireshark). To uninstall: Remove the build directory and packages. Always test in an isolated lab network.

Functions and Walkthroughs

Launch text mode: ettercap -T. Unified sniffing: Select interface and targets. ARP poisoning: ettercap -T -M arp:remote /target1/ /target2/. Sniff passwords: Enable plugins like autoadd or dns_spoof. Full options: ettercap --help. Operate only on networks with explicit authorization.

Detailed Tutorial: Performing ARP Spoofing MITM Attack (Ethical Lab Scenario)

This tutorial demonstrates a basic ARP poisoning MITM attack using Ettercap in text mode on a controlled lab network (e.g., virtual machines on the same subnet). This must be conducted only in environments you own or have explicit written permission to test. Unauthorized interception or modification of traffic is illegal and unethical.

  1. Prepare the environment: Ensure you are on the same LAN as the targets (e.g., victim at 192.168.1.50, gateway at 192.168.1.1). Enable IP forwarding if needed: echo 1 > /proc/sys/net/ipv4/ip_forward (or use sysctl -w net.ipv4.ip_forward=1).
  2. Launch Ettercap in text mode: ettercap -T -q (-q for quiet mode to reduce output noise).
  3. Scan for hosts: Press 's' to scan the local subnet or manually add targets. Use 'h' to view the host list.
  4. Add targets: Select the victim IP (e.g., 192.168.1.50) as Target 1 and the gateway/router IP (e.g., 192.168.1.1) as Target 2. Press 't' to set targets.
  5. Start ARP poisoning: Press 'p' to start the MITM attack (Ettercap begins ARP spoofing, redirecting traffic through your device).
  6. Sniff traffic: Ettercap automatically captures and displays packets. Use plugins (press 'o' to manage) such as 'autoadd', 'dns_spoof', or 'remote_browser' for advanced features. Observe credentials or HTTP data in real time.
  7. Monitor and interact: Watch for dissected protocols (e.g., HTTP POST data). For demonstration, visit an HTTP site on the victim machine and observe captured form submissions.
  8. Stop the attack: Press 'p' again to stop poisoning, then 'q' to quit Ettercap. Restore ARP tables naturally or manually on targets if needed.
  9. Clean up: Disable IP forwarding if enabled (echo 0 > /proc/sys/net/ipv4/ip_forward). Document observations responsibly for security reporting.

Expected results: Traffic redirection succeeds, allowing inspection of unencrypted data between targets. Common issues: Targets use HTTPS (limits visibility—demonstrate downgrade risks ethically); ARP cache persistence—wait or flush caches post-test. Use display filters or logs for analysis.
