Nemesis

Nemesis is a command-line packet injection framework that enables the creation and transmission of custom packets with complete control over protocol headers, options, payloads, and fragmentation. Originally developed by Mark Grimes and maintained sporadically by the community, Nemesis supports a broad range of protocols including Ethernet, IP, TCP, UDP, ICMP, IGMP, RIP, OSPF, BGP, and GRE. It is particularly useful for low-level network protocol testing, crafting malformed or non-standard packets, validating firewall and IDS behavior, testing protocol implementations, and simulating attack traffic in authorized security research and penetration testing environments.

Uses

Nemesis is employed in authorized network security testing to generate protocol-specific test packets, evaluate firewall rule processing, test IDS signature evasion techniques, perform path MTU discovery with custom fragmentation, simulate routing protocol advertisements, and validate TCP/IP stack robustness against malformed input. Example: Sending crafted OSPF hello packets with invalid checksums or BGP UPDATE messages with malformed attributes to observe target device behavior during controlled protocol fuzzing in a lab network.

Installation in Termux

Nemesis is not packaged in standard Termux repositories and requires compilation from source. Raw packet injection requires root privileges (use tsu or a rooted device). Installation steps:

pkg update && pkg upgrade -y
pkg install git build-essential libpcap-dev libdnet-dev libnet-dev -y
git clone https://github.com/dugsong/nemesis.git
cd nemesis
./autogen.sh
./configure --prefix=/data/data/com.termux/files/usr
make
make install
# Test: nemesis -V
# Note: If libdnet/libnet issues occur, install libdumbnet-dev and libnet1-dev equivalents or build them separately

Common issues: Raw socket permission denied—run with root (tsu -c nemesis ...); missing libraries—ensure libpcap-dev, libdnet-dev, and libnet-dev are installed; autogen failure—install autoconf/automake/libtool. To uninstall: Remove the cloned directory and run make uninstall if supported. Full packet injection capabilities typically require a rooted device or Kali NetHunter environment.

Functions and Walkthroughs

Basic TCP packet: nemesis tcp -S 192.168.1.100 -D 192.168.1.1 -p 80 -f S -c 1 (SYN packet). UDP with payload: nemesis udp -S 10.0.0.1 -D 8.8.8.8 -p 53 -P payload.bin. ICMP echo request: nemesis icmp -S 192.168.1.100 -D 192.168.1.1 -i 8 -c 5. OSPF hello packet: nemesis ospf -v -S 192.168.1.1 -D 224.0.0.5 -a hello. BGP UPDATE: nemesis bgp -v -S 172.16.1.1 -D 172.16.1.2 -a update -A 65001 -N 192.168.0.0/16. Always test on isolated lab networks with explicit written permission.

Detailed Tutorial: Testing Firewall Behavior with Custom TCP Packets (Ethical Lab Scenario)

This tutorial demonstrates using Nemesis to send specially crafted TCP packets to evaluate firewall filtering, state tracking, and IDS response in a controlled lab environment (e.g., a local network with a test firewall or target system you own). This procedure is permitted only on networks and systems you control or for which you have explicit written authorization. Unauthorized packet injection is illegal and may be interpreted as a network attack.

  1. Prepare environment: Ensure Nemesis is compiled and installed. Confirm root access (tsu or su). Identify target IP (e.g., firewall at 192.168.1.1) and test port (e.g., 80). Verify lab isolation and permission.
  2. Send standard SYN packet: Run tsu -c "nemesis tcp -S 192.168.1.100 -D 192.168.1.1 -p 80 -f S -c 1 -v". Observe response (SYN/ACK from open port, RST from closed, no reply from filtered).
  3. Test unusual flag combinations: Send SYN+FIN packet (invalid but sometimes bypasses naive filters): tsu -c "nemesis tcp -S 192.168.1.100 -D 192.168.1.1 -p 80 -f SF -c 1 -v"
  4. Send large-payload TCP packet: Test fragmentation handling or buffer issues: tsu -c "nemesis tcp -S 192.168.1.100 -D 192.168.1.1 -p 80 -d 1400 -c 1 -v"
  5. Inject packet with custom options (TCP timestamp): tsu -c "nemesis tcp -S 192.168.1.100 -D 192.168.1.1 -p 80 -f S -O timestamp=123456789 -c 1 -v"
  6. Send spoofed-source packet: Test anti-spoofing rules: tsu -c "nemesis tcp -a 8.8.8.8 -S 192.168.1.100 -D 192.168.1.1 -p 80 -f S -c 1 -v"
  7. Monitor and analyze: Use tcpdump/Wireshark in parallel on the target or another host to capture Nemesis-generated packets. Review firewall/IDS logs for accepted/rejected/dropped packets, alerts on malformed options, or anti-spoofing events.
  8. Document and recommend: Record packet types, flags, payloads, and target responses. Suggest improvements: strict stateful inspection, invalid flag drop rules, anti-spoofing ACLs, rate limiting, and logging of anomalous packets.

Expected results: Precise control over sent packets with observable target behavior (accepted, rejected, dropped, or unexpected resets), validating firewall/IDS configuration effectiveness. Common issues: Packets dropped silently—target silently discards; spoofing ineffective—requires root and proper egress filtering disabled in lab; interface errors—verify correct -d device flag if needed.
