hping3

hping3 is a command-line network tool capable of sending custom TCP/IP packets with full control over header fields, flags, payload, sequence numbers, source/destination ports, and transmission timing. Originally developed by Salvatore Sanfilippo (antirez), it functions as an enhanced packet generator, allowing users to craft arbitrary IP, TCP, UDP, ICMP, and raw packets. hping3 is widely used in ethical network testing for firewall/IDS rule validation, port scanning, path MTU discovery, TCP sequence prediction, fragmentation testing, denial-of-service simulation, and traceroute-style probing with custom protocols. It remains one of the most flexible low-level packet crafting utilities available.

Uses

hping3 is employed in authorized penetration testing and network security assessments to test firewall filtering rules, evaluate IDS/IPS detection logic, perform stealthy port scanning, simulate fragmented or malformed traffic, measure latency/jitter, and validate TCP/IP stack behavior. Example: Sending crafted TCP packets with specific flags (SYN/ACK/RST) or payloads to determine firewall statefulness, or using ICMP/UDP mode to perform traceroute with arbitrary protocol evasion in a controlled lab environment.

Installation in Termux

hping3 is available in Termux repositories or can be compiled from source. Raw socket access typically requires root privileges (use tsu or a rooted device). Installation options:

pkg update && pkg upgrade -y
# Option 1: Install from Termux repository (recommended)
pkg install hping3 -y
# Option 2: Compile from source (for latest version or custom build)
pkg install git build-essential libpcap-dev tcl-dev -y
git clone https://github.com/antirez/hping.git
cd hping
./configure
make
make install
# Test: hping3 --version

Common issues: Permission denied on raw sockets—run with root (tsu -c hping3 ...); libpcap errors—ensure libpcap-dev is installed; missing tcl—some features (scripting) require tcl-dev. To uninstall: Remove the package or compiled binary. Full packet-crafting capabilities often require a rooted device or Kali NetHunter.

Functions and Walkthroughs

Basic SYN scan: hping3 -S 192.168.1.1 -p 80 -c 1. Flood mode: hping3 --flood -S 192.168.1.1 -p 80. Custom payload: hping3 -d 120 -E payload.txt 192.168.1.1. Traceroute with TCP: hping3 --traceroute -V -1 8.8.8.8 (ICMP mode). Spoof source IP: hping3 -a 1.2.3.4 192.168.1.1. Fragmentation: hping3 --frag 192.168.1.1. Always test on networks and systems you own or have explicit written permission to probe.

Detailed Tutorial: Firewall Rule Testing with Custom TCP Packets (Ethical Lab Scenario)

This tutorial demonstrates using hping3 to send crafted TCP packets and observe firewall/IDS behavior in a controlled lab environment (e.g., a local network with a test firewall or target system you own). This procedure is permitted only on networks and systems you control or for which you have explicit written authorization. Unauthorized packet injection or probing is illegal and may be interpreted as a network attack.

  1. Prepare environment: Ensure hping3 is installed and you have root access (tsu or su). Identify your interface and target IP (e.g., firewall/gateway at 192.168.1.1). Confirm lab isolation and permission.
  2. Test basic connectivity: Run tsu -c "hping3 -c 3 192.168.1.1" to send three default TCP packets and observe responses (SYN/ACK, RST, or no reply).
  3. Send SYN packets to specific port: tsu -c "hping3 -S -p 80 -c 5 192.168.1.1". This sends five TCP SYN packets to port 80. Expected: open port returns SYN/ACK, closed returns RST, filtered returns nothing or ICMP unreachable.
  4. Test firewall filtering: Send packets with unusual flags: tsu -c "hping3 -S -p 80 --tcp-timestamp 192.168.1.1" (SYN with TCP timestamp option) tsu -c "hping3 -F -p 80 192.168.1.1" (FIN packet – may bypass stateless filters) tsu -c "hping3 -c 1 -d 100 -E /dev/zero 192.168.1.1 -p 443" (large null-payload packet to test fragmentation handling)
  5. Perform stealth scan simulation: Send SYN packets slowly with randomized source ports: tsu -c "hping3 --rand-source -S -p 22,80,443 -i u10000 192.168.1.1" (10 ms interval, random source IP spoofing – requires root)
  6. Observe and analyze: Monitor target firewall logs, IDS alerts, or system responses. Use Wireshark/tcpdump in parallel to capture hping3 traffic for verification. Note which packets are allowed/blocked and how the target responds (RST vs. drop vs. ICMP).
  7. Document and report: Record packet types, flags, responses, and firewall behavior. Recommend improvements: stateful inspection, rate limiting, anomaly detection, and logging of malformed or suspicious packets.

Expected results: Clear feedback on firewall filtering behavior (allowed, blocked, dropped, or reset) for various packet types, helping validate rule completeness and state tracking. Common issues: No response—target may silently drop; spoofing not working—requires root and proper interface configuration; high-rate packets dropped—network congestion or rate limiting on path.
