Bettercap

Bettercap is a powerful, modern, and actively maintained Swiss-army-knife framework for network reconnaissance, man-in-the-middle (MITM) attacks, traffic manipulation, and wireless auditing. Written in Go by Simone Margaritelli (evilsocket), it supports Ethernet, Wi-Fi, Bluetooth Low Energy (BLE), and HID attacks, with an extensive module ecosystem for sniffing, spoofing, proxying, injection, and more. Bettercap is widely regarded as a more performant, modular, and up-to-date alternative to legacy tools such as Ettercap, arpspoof, and dnsspoof. It is commonly used in authorized red team engagements, network security assessments, and wireless penetration testing.

Uses

Bettercap is employed in ethical penetration testing to perform network reconnaissance, conduct MITM attacks (ARP/DNS spoofing), intercept and manipulate cleartext and SSL traffic (via HTTPS downgrade or HSTS bypass techniques), sniff credentials, inject JavaScript, and audit wireless networks. Example: Performing ARP spoofing on a local network to intercept HTTP/HTTPS traffic, downgrade connections, and harvest credentials during an authorized red team exercise in a controlled lab environment.

Installation in Termux

Bettercap provides pre-built static binaries for Android/arm64, making installation straightforward in Termux. Root is required for most network manipulation modules (use tsu or rooted device). Steps:

pkg update && pkg upgrade -y
pkg install wget -y
# Download latest stable release for arm64 (check https://www.bettercap.org/releases/ for current version)
wget https://github.com/bettercap/bettercap/releases/download/v2.32.0/bettercap_linux_arm64_v2.32.0.zip
unzip bettercap_linux_arm64_v2.32.0.zip
chmod +x bettercap
mv bettercap $PREFIX/bin/
# Optional: Install capstone for disassembly support
pkg install capstone -y
# Test: bettercap -h
# Run with root: tsu -c bettercap

Common issues: Permission denied on interfaces—run with root (tsu/su); no monitor mode for Wi-Fi—requires compatible USB adapter and drivers; capstone missing—some modules require it. To uninstall: Remove the binary from $PREFIX/bin. Full capabilities (especially wireless and packet injection) often require Kali NetHunter or a rooted device with appropriate hardware.

Functions and Walkthroughs

Launch interactive mode: bettercap -iface wlan0. Web UI: bettercap -iface eth0 -caplet http-ui (access http://:80). Key commands: net.probe on, net.show, arp.spoof on, set arp.spoof.targets 192.168.1.50, http.proxy on, https.proxy on, set http.proxy.sslstrip true. Modules: wifi.recon on, ble.recon on, hid.attack, dns.spoof. Always operate on networks and systems you own or have explicit written permission to test.

Detailed Tutorial: ARP Spoofing + HTTP/HTTPS Traffic Interception (Ethical Lab Scenario)

This tutorial demonstrates using Bettercap to perform ARP spoofing and intercept HTTP/HTTPS traffic in a controlled lab environment (e.g., a local network with test VMs or devices you own). This procedure is permitted only on networks and systems you control or for which you have explicit written authorization. Unauthorized interception or manipulation of network traffic is illegal.

  1. Prepare environment: Ensure Bettercap is installed and you have root access (tsu or su). Identify your interface (e.g., wlan0) and target IP (e.g., victim 192.168.1.50, gateway 192.168.1.1). Enable IP forwarding if needed: echo 1 > /proc/sys/net/ipv4/ip_forward.
  2. Launch Bettercap: Run tsu -c "bettercap -iface wlan0" to start interactive session on the desired interface.
  3. Discover hosts: Type net.probe on then net.show to populate the host list. Identify victim and gateway MAC/IP pairs.
  4. Configure ARP spoofing: Set targets set arp.spoof.targets 192.168.1.50 (or comma-separated list), enable full-duplex spoofing set arp.spoof.fullduplex true, and start arp.spoof on. Bettercap begins poisoning ARP tables.
  5. Enable proxy modules: Start HTTP proxy http.proxy on, HTTPS proxy https.proxy on, and optional SSL stripping set http.proxy.sslstrip true (attempts to downgrade HTTPS to HTTP where possible). Optionally enable JavaScript injection: js.inject on.
  6. Monitor intercepted traffic: Use net.sniff on to see cleartext credentials, cookies, and POST data. Access web UI (http://localhost:80) for graphical view of events, packets, and hooked browsers (if using BeEF integration).
  7. Execute advanced actions (lab only): Try dns.spoof on with custom spoof records, or http.proxy.injectjs to insert client-side payloads. Observe victim behavior when browsing HTTP/HTTPS sites.
  8. Stop and clean up: Type arp.spoof off, http.proxy off, https.proxy off, net.sniff off, then quit with quit. Disable IP forwarding if enabled. Document observations responsibly and recommend defenses: static ARP entries, dynamic ARP inspection, HTTPS enforcement (HSTS), certificate pinning, and endpoint monitoring.

Expected results: Successful ARP poisoning, traffic redirection through your device, and visible interception of cleartext credentials or HTTP downgrades. Modern targets with HSTS/HTTPS-only may limit visibility unless downgrade techniques succeed. Common issues: Targets detect spoofing—reduce spoofing rate; no HTTPS downgrade—modern browsers enforce HSTS; wireless issues—requires monitor mode adapter for Wi-Fi modules.
